Part I · Author

Author it
the right way.

Anthropic's documented best practices for writing a Skill that Claude can discover and use effectively. Everything here maps directly to the official guide.

Source · Skill authoring best practices + Agent Skills overview

01AUTHOR

Concise is key — assume Claude is smart

The context window is a public good. Every token in SKILL.md competes with conversation history and other Skills. Only add context Claude doesn't already have — challenge each line: "Does it really need this? Does this paragraph justify its token cost?"

Token budgetKeep the SKILL.md body under 500 lines; split into files past that.
conciseness<500 linescontext budget
02AUTHOR

Build evaluations first

Create evals before writing extensive documentation — it ensures the Skill solves real problems, not imagined ones. Run Claude on representative tasks without the Skill to find the gaps, build at least three scenarios that test them, baseline, then write the minimal instructions that pass.

{
  "skills": ["pdf-processing"],
  "query": "Extract all text and save to output.txt",
  "expected_behavior": ["reads PDF", "all pages", "saves output.txt"]
}
eval-driven≥3 scenariosbaseline first
03AUTHOR

Write the description in third person

The description is how Claude picks one Skill out of 100+. It must say what it does and when to use it, packed with concrete trigger terms — and always in third person, since it's injected into the system prompt.

✓ description: Extract text and tables from PDFs, fill forms. Use when working with PDF files or forms.
✗ "I can help you process PDFs"   # wrong POV
third persontrigger termswhat + when
04AUTHOR

Name with gerunds, lowercase-hyphens

Consistent names make Skills easy to reference and reason about. Prefer gerund form — processing-pdfs, analyzing-spreadsheets. The name field must be lowercase letters, numbers and hyphens only, max 64 chars, and may not contain reserved words ("anthropic", "claude").

Avoid vague names like helper, utils, or tools.

gerund formlowercase-hyphensno reserved words
05AUTHOR

Use progressive disclosure

SKILL.md is a table of contents, not the encyclopedia. Bundle detail in separate files that load only when needed. Keep references one level deep from SKILL.md (Claude may only partially read nested files), and give reference files over 100 lines a table of contents.

Three load levelsMetadata always loaded → SKILL.md when triggered → bundled files & scripts only when read. Scripts execute without entering context.
progressive disclosureone level deepToC over 100 lines
06AUTHOR

Set the right degrees of freedom

Match specificity to the task's fragility. High freedom (text guidance) when many approaches work — e.g. code review. Medium (parameterized scripts) when a preferred pattern exists. Low (exact commands, "do not modify") when operations are fragile and order matters — e.g. database migrations.

The bridge analogyNarrow bridge over a cliff → exact guardrails. Open field → general direction, trust Claude to route.
degrees of freedomhigh / medium / lowfragility-matched
07AUTHOR

Provide workflows & feedback loops

Break complex tasks into clear sequential steps, and for multi-step work give Claude a checklist it can copy and tick off. Build in validator loops — run validator → fix errors → repeat — and create verifiable intermediate outputs (plan → validate → execute) for batch or destructive operations.

checklistsvalidate → fix → repeatplan-validate-execute
08AUTHOR

Test across all models you'll use

A Skill is an addition to a model, so its effectiveness depends on that model. Test with every model you plan to run it on. What works perfectly for Opus may underspecify for Haiku; what's crisp for Haiku may over-explain for Opus.

multi-modelHaiku · Sonnet · Opusreal scenarios
09AUTHOR

Avoid time-sensitive info; stay consistent

Don't bake in "before August 2025…" — it goes stale. Put deprecated guidance in a collapsed "old patterns" section instead. And choose one term per concept and use it throughout (always "field", always "extract") — inconsistent vocabulary confuses the model.

no time-sensitiveold-patterns sectionconsistent terms
10AUTHOR

Scripts that solve, don't punt

When bundling code, handle errors explicitly rather than failing and leaving Claude to figure it out. No "voodoo constants" — justify every magic number. Use forward-slash paths (works everywhere), don't list five library options when one default will do, and reference MCP tools by fully-qualified name (ServerName:tool_name).

solve don't puntno voodoo constantsforward slashes
Part II · Extend

Beyond a
single file.

Once the basics are solid: decide who pulls the trigger, push heavy work into isolated subagents, compose multi-stage workflows, and reach for hooks when instructions alone can't guarantee a thing happens.

Source · Skills · Subagents · Hooks

11EXTEND

Control who invokes a skill

By default skills are model-invoked — Claude auto-triggers them from the description. For destructive or explicit-only actions, set disable-model-invocation: true so the skill runs only when the user types /skill-name. Capture input with the $ARGUMENTS placeholder.

---
description: Deploy to production
disable-model-invocation: true   # user-only, via /deploy
---
model vs user invoked/skill-name$ARGUMENTS
12EXTEND

Offload to subagents for context & cost

A subagent is a specialized worker that runs in its own context window with a custom system prompt, restricted tools, and independent permissions — it does the noisy work (search, logs, exploration) and returns only a summary. Define one in .claude/agents/ (or a plugin's agents/) and Claude delegates by matching its description.

---
name: test-runner
description: Runs and triages the test suite
tools: Bash, Read, Grep      # least privilege
model: haiku                 # cheap for grunt work
---

Use them to preserve your main context, enforce tool limits, and route grunt work to faster, cheaper models.

own context window.claude/agents/tools + model
13EXTEND

Compose when one skill isn't enough

A single SKILL.md handles one job; real tasks often chain several. Compose the primitives: a skill supplies the procedure, subagents run heavy or isolated stages in their own context, and hooks gate the transitions between them. That separation is what makes long workflows reliable instead of brittle.

Inside each stage, lean on the step-level patterns from Part I · 07 — checklists, validator loops, plan→validate→execute — rather than repeating them.

PatternOrchestrator skill → a subagent per stage → a hook at each gate. Keep every stage independently verifiable.
compositionskill + subagent + hookisolated stages
14EXTEND

Enforce with hooks what instructions can't

CLAUDE.md and skills are context, not guarantees — Claude reads them but may not comply perfectly. For anything that must happen every time (lint before commit, block a protected path, auto-sync the repo), use a PreToolUse / PostToolUse hook: a shell command fired at a fixed lifecycle event, run by the harness regardless of what Claude decides.

Hooks are the deterministic backstop beneath your CLAUDE.md and ops conventions (Parts III–IV) — they're how "always" actually becomes always.

PreToolUse / PostToolUsedeterministicthe enforcement layer
Interlude · Capacity

How many Skills
before it strains?

There's no documented hard cap — Claude selects "from 100+ Skills." The real limit is a character budget for the skill metadata in the system prompt. Past it, the descriptions of your least-used Skills get trimmed or dropped — silently.

Comfortable
< 30

Plenty of headroom. Just keep each description distinct and keyword-rich.

Tighten up
40–60

Keep descriptions under ~150 chars, trigger terms in the first ~50. Crowding starts to bite.

Crowded
60+

Under ~130 chars each; expect least-used skills to get trimmed. Split project vs. global scope.

Observed: 63 skills installed~16k char budget
42 visible to Claude 21 silently hidden (~33%)

Budget block ≈ 15.5–16k chars (~1% of context window); names always shown, descriptions dropped least-used-first. Figures are community-measured, not official — see claude-code #13099. Bottom line: plan for ~40–60 well-described Skills; description quality matters more than count (see Part I · 03–04).

Part III · Configure

stop repeating.

Skills don't live in a vacuum. CLAUDE.md gives Claude the standing context it needs, and permissions decide what runs without a prompt. Both are layered — broad to specific — so a few files set the tone for every session.

15CONFIGURE

Build CLAUDE.md for what you'd re-explain

CLAUDE.md is loaded every session, so put facts there, not procedures: build/test commands, conventions, project layout, "always do X" rules. Run /init to generate a first draft from your codebase, then refine. Write the thing you keep re-typing into chat.

Be specific and concise — "Use 2-space indentation" beats "format nicely." Keep it under ~200 lines; longer files dilute adherence. Multi-step or path-specific guidance belongs in a skill or a .claude/rules/ file instead.

TipPull in external files with @path imports (e.g. @AGENTS.md) — they expand into context at launch, max four hops deep.
/init<200 linesfacts not procedures
16CONFIGURE

Know the hierarchy — it stacks, it doesn't fight

CLAUDE.md files don't override each other; they're concatenated from broadest (managed → user → project → local) down to most specific, so instructions closest to where you launched Claude are read last. Files in subdirectories load on demand when Claude touches files there.

Permissions follow a precedence order instead: managed → CLI → local → project → user. They merge across layers, but deny always wins — a project denial overrides a user allow. Use /memory and /permissions to see exactly what's loaded and why.

concatenated, not overriddendeny winssubdir = on demand
17CONFIGURE

Grant permanent permission so it stops asking

To stop the per-use prompt, add an allow rule to settings.json. Allow a skill by name, and allow the tools it runs (Bash patterns support *). The fastest path: when a prompt appears, choose "Yes, and don't ask again" — it writes the rule to .claude/settings.local.json for you.

{
  "permissions": {
    "allow": [
      "Skill(code-review)",
      "Bash(npm run test:*)",
      "Bash(git status)"
    ]
  }
}

Pick the layer by reach: ~/.claude/settings.json for all your projects, .claude/settings.json to share with the team (commit it), or .claude/settings.local.json for just you (gitignored).

Why it still asksMost repeat prompts come from the Bash commands a skill runs, not the skill itself — so allowlist those Bash(...) patterns, not just Skill(name).
permissions.allowSkill(name)don't ask again
Part IV · Maintain

Keep it
trustworthy.

Practices for keeping a growing Skill library reliable over time. These are general software-engineering conventions — high value, but the part of this guide that is not in Anthropic's official spec.

Heads-up: none of these are required by the SKILL.md format. Notably, there's no version field in the frontmatter and Skills don't auto-sync across surfaces — so treat this part as team convention, not platform features.
18MAINTAIN

One source of truth + auto-sync

A Skill scattered across laptops and "final_v2" folders drifts until no one knows which copy is real. Make the git repo canonical and automate the sync into wherever Skills load, so the deployed Skill is the committed Skill. Pull, don't paste.

# post-merge: sync canonical skills into the live dir
rsync -a --delete repo/skills/  ~/.claude/skills/
# one direction only — repo is upstream
source of truthauto-syncteam convention
19MAINTAIN

Track a version

The spec only requires name + description, but I add a SemVer-style version (in frontmatter or a sidecar file) so rollbacks are possible and bug reports are legible — "broke in 2.1.0" beats "broke sometime." MAJOR for breaking behavior, MINOR for capability, PATCH for fixes.

semverrollbacknot in spec
20MAINTAIN

Keep a changelog

Pair every version bump with a human-readable "what changed and why," grouped Added / Changed / Fixed / Deprecated. One scannable file beats reading diffs to reconstruct history — especially when teammates depend on the Skill.

## [2.3.1] — 2026-06-10
### Fixed false trigger on docs-only PRs
### Changed tightened description trigger terms
changelogkeep-a-changeloghistory
21MAINTAIN

Gate with CI validation

Don't trust frontmatter by eye. Run a CI check on every PR that validates the YAML schema and required fields, verifies internal links resolve one level deep, and runs the eval suite. The official docs ship a pre-share checklist — I just automate it.

✓ frontmatter valid · ✓ links resolve
✓ version bumped + changelog entry
✓ 24/24 eval cases pass
CI gateschema lintautomate the checklist
22MAINTAIN

Audit for least privilege

A Skill is a security surface — even your own. Keep bundled scripts narrow and idempotent, avoid surprise network calls, and review what each Skill can actually reach before it ships. (Adopting other people's skills is its own discipline — see Part V · 27.)

least privilegeaudit bundled codeidempotent
23MAINTAIN

Assign owners, deprecate gracefully

Every Skill needs a name attached — someone who answers "is this still right?" When one outlives its purpose, deprecate loudly: note it in the changelog, point to the replacement, give a migration window. (The docs' "old patterns" section from Part I · 09 is the in-file half of this.)

LifecycleBorn → owned → versioned → deprecated → removed. Never just abandoned.
ownershipgraceful deprecationmigration notes
Part V · Distribute

Ship it past
your own laptop.

A skill only earns its keep when others can install it safely. Package it, distribute it, keep it portable across surfaces — and treat every third-party skill as code you're choosing to run.

Source · Plugins · Marketplaces · Skills overview

24DISTRIBUTE

Package skills into plugins

Start standalone in .claude/skills/ for fast iteration; convert to a plugin when you're ready to share. A plugin is a self-contained directory with a .claude-plugin/plugin.json manifest that can bundle skills/, agents/, hooks/, and MCP servers. Skills get namespaced as /plugin-name:skill to avoid collisions.

{
  "name": "pr-toolkit",
  "description": "PR review + changelog skills",
  "version": "1.2.0",
  "author": { "name": "you" }
}

Test locally with claude --plugin-dir ./pr-toolkit and hot-reload with /reload-plugins.

plugin.jsonnamespaced skills--plugin-dir
25DISTRIBUTE

Distribute through marketplaces

A marketplace is just a git repo with a .claude-plugin/marketplace.json catalog. Users run /plugin marketplace add <repo> then /plugin install. Keep it private for internal-only skills, or submit to Anthropic's claude-community marketplace for public reach.

Set an explicit version so installs are stable — omit it and every git commit counts as a new version. Always run claude plugin validate before publishing.

Versioned releasesThis is where Part IV's SemVer + changelog finally pay off — pinned versions give users controlled, reviewable updates.
marketplace.json/plugin installplugin validate
26DISTRIBUTE

Plan for cross-surface portability

Skills do not sync across claude.ai, the Claude API, and Claude Code — each surface is managed separately, with different runtime limits (the API has no network or package install; Claude Code has full access). Don't assume "uploaded once = available everywhere."

The format itself is portable: skills follow the Agent Skills open standard, so the same SKILL.md travels even when the install path doesn't. Keep your repo canonical (Part IV · 18) and publish outward to each surface.

no auto-syncopen standardper-surface limits
27DISTRIBUTE

Treat skills as a supply chain

Skills carry instructions and executable code, so a malicious one can misuse tools, exfiltrate data, or smuggle prompt-injection through a fetched URL. Install only from sources you trust, and audit every bundled file — SKILL.md, scripts, hooks — before adopting a community plugin.

Be especially wary of skills that pull external content at runtime; even a trustworthy skill can be compromised through its dependencies. The rule: treat installing a skill like installing software.

Before you adoptRun claude plugin validate, read the scripts, and check for network calls that don't match the stated purpose.
audit before installprompt-injectiontrusted sources only
Appendix · Starter

Copy. Rename.
Ship.

A canonical SKILL.md that already follows the rules above — concise body, third-person trigger-rich description, progressive-disclosure references, and a validator-loop workflow. Drop it in .claude/skills/your-skill-name/ and edit.

Directory scaffold

your-skill-name/
├── SKILL.md          # entry point, <500 lines
├── reference/
│   └── details.md    # loaded on demand
├── examples/
│   └── basic.md      # input → output pairs
└── scripts/
    └── validate.py   # executed, not read
SKILL.md
---
name: your-skill-name          # lowercase-hyphens, gerund form
description: Does X for Y. Use when <trigger terms a user would type>.
---

# Your Skill Name

## Quick start
The single most common path — one minimal, working example.
Assume Claude already knows the basics; add only what it doesn't.

## Workflow
Copy this checklist and check off each step:

- [ ] Step 1: <action>
- [ ] Step 2: <action>
- [ ] Step 3: validate — run `scripts/validate.py`, fix errors, repeat

## Reference  # one level deep, loaded only when needed
- Advanced options → reference/details.md
- Worked examples  → examples/basic.md

## Rules
- ALWAYS <hard requirement>.
- NEVER <anti-pattern>.
